The shift that changes everything

You're not coding. You're designing systems. Every prompt is a brief you're giving to an engineer. Every bug is feedback from a user. Every edge case is a feature waiting to fail. Once you think this way, everything else follows.

01 · The mindset

Before you build anything

The mistakes I made when I started weren't technical. They were strategic. Here's what I wish I'd known.

01

Plan before you build

AI doesn't plan ahead unless you do. If you just say "build this," it starts at the top and skips everything else. Prompt together first — see the structure, make edits, then implement. Every time I skipped this step, it cost me.

02

Give context, always

AI doesn't know your context. What page are you on? Which user sees this? Which component is affected? The more context you give, the less guessing it does. Take the extra 30 seconds. It saves hours.

03

Design for real life, not the happy path

Every feature you build has five states. If you only build the success state, you've built half a feature — and the other half will humiliate you in front of a user.

Build every feature with 5 states: Default · Loading · Success · Error · Empty
04

Git is your undo button

Even if you're not technical, set up Git from day one. Use branches for each feature, commit often, push to GitHub. AI makes mistakes. So do you. Git means you can always go back.

05

Track what users do

If you don't log user events, you're building blind. Create a user_events table. Track login, key actions, form submissions. Ask yourself: can I answer "what did the user do?" If not, you're guessing.

06

Test like a human, not a developer

Log in. Log out. Go through the whole flow. Try it on mobile. Pretend it's your first time. Then try to break it — refresh mid-form, submit weird data, remove required fields. If it handles all of that, you're done.

02 · The prompt formula

Structure every prompt like a brief

If you can't explain it clearly, the AI won't build it correctly. These six elements work at the beginning of a project and at the feature level as you go.

WHAT
What you're building
e.g. "A recipient selection screen"
WHO
Who it's for
e.g. "A lawyer under time pressure at 10pm"
PURPOSE
What it should do
e.g. "Fast validation, not a complete form"
DESIGN
What it should look like
e.g. "Here's a screenshot of the competitor"
STATES
All conditions to handle
e.g. "Loading, empty, error, full"
CONSTRAINTS
Technical limits
e.g. "Must work on mobile, RLS enabled"

Key prompts · copy and use

When you're stuck in an error loop
"Review the full code. List all issues you found. Suggest a fix for each one and share them with me. Then suggest a fix that addresses all the issues together."
Build an auth flow
"Create an authentication flow using Supabase. Include:
- Sign-up: email + password + name
- Login with email and password
- Forgot password flow
- Redirect: if onboarding incomplete → onboarding. Otherwise → dashboard."
Multi-tenant data structure
"Build a multi-tenant structure where each record is tied to a client_id. Allow an admin user to switch between clients using a dropdown, and make sure only the selected client's data is shown. Include role-based access logic and data filtering."
File uploads
"Set up a file upload system using Supabase Storage. Allow users to upload documents and save metadata in a 'documents' table with: user_id, file_url, type, label, created_at. Include functionality to update the label and delete the file."
Security — Row-Level Security
"Enable Row-Level Security in Supabase for all sensitive tables. Write RLS policies so users can only access rows where client_id matches their assigned client. For admin users, allow access to multiple clients. Add a 'role' field to differentiate user types."
Wrap a component with all 5 states
"Wrap this component with loading, success, error, and empty states. Add helpful CTAs for each state. If the API fails, show a retry button and a friendly error message. On mobile, show rows as cards instead of a table."

05 · The backend blueprint

How I built a real backend

When people read my first guide, they asked: "OK but how do I build something where users can log in, upload files, and pay?" This is the answer. Six building blocks that work together.

01 · Auth

Login, sign-up, forgot password

Use Supabase's built-in auth. Connect it in Lovable by pasting your project URL and key. You get sign-up, login, forgot password, and redirect logic out of the box. First-time users go to onboarding. Everyone else goes to their dashboard.

02 · Multi-tenant

One app, many clients

Every record in your database gets a client_id. Admins can switch between clients via a dropdown. Users only see data for their current client. I tested this with mock data first — two fake clients with completely separate records — to catch cross-client bugs early.

03 · Payments & Balances

Track every cent

Build a balances table (user_id, currency, amount) and a transactions table (user_id, amount, currency, recipient_id, timestamp, status). Status starts as pending, moves to completed or failed. When completed, subtract from balance. This becomes the backbone of the product.

04 · File Uploads

Documents, IDs, contracts

Uploads go to Supabase Storage. Each file gets metadata in a documents table: user_id, file_url, type, label, created_at. Users can rename and delete files. All paperwork, secured, organized, in the app.

05 · Security

Row-Level Security

Enable RLS on every table. Users can only read rows where client_id matches theirs. Admins get access to their assigned clients only. Add a role field to users to differentiate. If you handle money or personal data, this is not optional.

06 · Third-party integrations

Edge functions for everything else

Need to send emails, post to Slack, trigger an automation? Use Supabase Edge Functions. Small backend scripts that run on events. Store API keys as environment variables — never expose them in the frontend. This is how I connected to Resend, payment APIs, and more.
